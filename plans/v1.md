# agent-commander v1 plan

## Context

`agent-commander` (binary: `ac`) is a greenfield project at `~/proj/ac`. **Source is modern C** (C23 opportunistically; C17 baseline). **Build is `zig build` driving `zig cc`** — single-tool, free cross-compilation to Linux/macOS/Windows.

`ac` is an **attended-mode dashboard** for running several AI agents (Claude Code, Codex) at once. The UX is built around the way the user actually works with agents: fire several off, then *watch* what they're doing — not stare at one terminal each.

The whole UI revolves around three views:

1. **Dashboard (default)**: a row of "agent buttons" at the top (one per running agent, each showing live status — `thinking`, `tool: ripgrep`, `awaiting input`, `done`, `error`) plus a unified, interleaved **activity log** below: every tool call from every agent in chronological order (`12:34:56  cc-1 → ripgrep "TODO" src/  (0.3s, 12 matches)`). This is what the user looks at while agents grind. No agent's output dominates; nothing is missed.
2. **Drilldown chat (on demand)**: select an agent button → fullscreen chat with that one agent. Conversation, tool-call accordions, input box. `Esc` returns to dashboard.
3. **Curation (`Ctrl+G` from drilldown)**: `git add -p`-style hunk picker over the focused agent's transcript. Accepted hunks accumulate in a shared **clipboard**. `Ctrl+P` pastes the clipboard as input into any agent.

`ac` itself contains no LLM brain. The user is the orchestrator. `ac`'s value-add: aggregate status + log across many agents, frictionless drilldown, user-mediated context routing.

Constraints fixed by the user (in order of decision):
- **Source: modern C.** C23 where supported, C17 baseline.
- **Build: `zig build` / `build.zig`** driving `zig cc`. No CMake. Pin Zig version in `.zigversion`.
- **Cross-platform: Linux, macOS, Windows 10+.** Agent I/O is plain pipes (cross-platform trivially); a separate PTY layer behind `pty.h` is reserved for the deferred `sh` backend.
- **No daemon for v1.** Single binary; sessions die with the terminal.
- **No LLM brain in `ac` for v1.** No internal `claude -p` driver. User has no Anthropic API key — designs that need one are off the table.
- **UX = dashboard + drilldown, NOT tiled panes.** This is the explicit user choice.
- Agent backends: **Claude Code** (`cc`) and **Codex** (`codex`) — both run in their stream-json interactive modes so `ac` can parse tool calls and progress events.
- Style: project's `CLAUDE.md` references the Nx C++ style guide. We're in C, so it doesn't apply literally. Apply *in spirit* (naming clarity, comment density, error-handling discipline).

### AgentSession

One per running agent. Owns:

- A child process spawned with **plain pipes** (no PTY): stdin / stdout / stderr.
  - cc: `claude --output-format stream-json --input-format stream-json --verbose --permissionMode dontAsk <prompt>` (resumable across messages within the single child lifetime).
  - codex: equivalent flags (verify exact form during step 1 spike).
- The child is spawned with **a configurable `cwd`** — defaults to `ac`'s own cwd, overridable per spawn (`:spawn cc --cwd /path`). This is what makes cc/codex's normal config discovery (`CLAUDE.md` walk, `AGENTS.md` walk) work transparently — `ac` does *not* manage agent instructions in v1; it just gets out of the way.
- A **stdout reader thread** that parses NDJSON line-by-line into `AgentEvent`s and pushes them to the central `EventBus`.
- A **transcript** — append-only structured log of `{role, content_blocks, tool_calls, tool_results}` reconstructed from the parsed events; this is what the drilldown view renders.
- A **status** field (atomic): `idle`, `thinking`, `tool(name)`, `streaming`, `awaiting_input`, `done`, `error`.

Sending a user message: serialize the message as the appropriate JSON shape and write to the child's stdin.

**Agent configuration in v1: inherit, don't manage.**
- cc picks up the user's `~/.claude/CLAUDE.md`, project-local `CLAUDE.md` (walked up from cwd), `~/.claude/skills/`, and `.claude/skills/` automatically. `ac` does nothing for this — spawning with the right cwd is enough. The user explicitly chose this for v1: *"just use my existing claude skills when invoke cc"*.
- codex similarly walks for `AGENTS.md` and reads its own config dirs.
- This means `~/proj/ac/CLAUDE.md` (which references the Nx C++ style) is *for whoever is editing ac itself* — it does not influence cc children spawned *by* ac while working on other projects, because those children run with their target project's cwd.

**v2 direction (deferred):** `ac` owns a unified per-user "agent instructions" tree (e.g., `~/.config/ac/agents.d/`). On spawn, `ac` materializes the right format for the chosen backend into a temp/overlay location: `CLAUDE.md` for cc, `AGENTS.md` for codex. One source of truth, N projection formats. v1 deliberately punts on this so we don't bikeshed the canonical schema before the rest works.

### EventBus / Activity log

A central, mutex-guarded ring of `LogEntry`:

```
LogEntry {
  timestamp;
  agent_id;                   // "cc-1", "codex-2", ...
  kind;                       // tool_call | rate_limit | hook | status | error
  oneline;                    // pre-rendered summary for compact view
  tool_call_id;               // null unless kind == tool_call
  parent_tool_use_id;         // null unless this tool call ran inside a sub-agent (Task)
  transcript_ref;             // {agent_id, transcript_index} — full payload lives there
}
```

Reader threads append here *in addition to* updating their agent's transcript; full payloads (args, output text) are not duplicated into the ring — the inspector follows `transcript_ref` to fetch them on demand. Keeps the ring small and fast.

The dashboard's center pane renders the tail of this ring in **compact mode** (one line per entry). Hitting **F7 / `Tab` from the activity log** flips into the **Log Inspector** described below.

### Log Inspector (expanded log mode)

Entered with **F7** (or `Enter` on a focused log entry) from the dashboard. Same data, expanded layout — each tool-call entry shows args and output inline, fetched from the transcript via `transcript_ref`:

```
12:34:56  cc-1 → ripgrep "TODO" src/                    (0.3s, 12 matches)
          ├─ args: { pattern: "TODO", path: "src/" }
          └─ output:
               src/main.c:42: // TODO: handle EOF
               src/parser.c:118: // TODO: split modes
               ...
12:34:58  cc-1 → Read /home/.../config.json              (0.05s, ok)
          └─ output: { ... 4 KiB ... }
```

**Navigation — categorical jump keys** (lnav-style `e/E`, `w/W`, `o/O` analogs adapted to our typed entries; `j`/`k` and arrows still scroll line-by-line):

| keys | jump to |
|---|---|
| `e` / `E` | prev/next **error** entry |
| `t` / `T` | prev/next **tool call** (skip rate_limit, hook, status) |
| `r` / `R` | prev/next **rate_limit_event** |
| `a` / `A` | prev/next entry from **same agent** as cursor |
| `n` / `N` | prev/next entry with **same tool name** |
| `p` / `P` | prev/next entry with **same `parent_tool_use_id`** (sibling sub-agent calls) |
| `c` | toggle to the focused entry's correlation pair (started ↔ finished) |
| `PgUp`/`PgDn`, `Home`/`End` | page / top / bottom |
| `Space` | collapse/expand focused entry's output (per-entry override) |

**Selection + yank.**
- `v` enter visual selection (anchor at focused entry); `j/k` or arrows extend.
- `y` yank selected outputs into **ac's clipboard** (origin marker `log:cc-1:tool#42`; subsequent `Ctrl+P` from a drilldown pastes them as input).
- `Y` yank to **system clipboard** (OSC 52 by default; falls back to `xclip`/`pbcopy`/`clip.exe`).
- `Esc` exit selection if active, else close Inspector.

**Filtering — lnav-mirrored filter stack.** A filter panel toggles via `\`; filters apply as a stack of typed predicates (no regex DSL needed for common cases — fields are typed). Status line when filters active: `2 of 3 filters · 28,401 entries hidden`.

Filter creation hotkeys (mnemonics match lnav's `Create:` line one-to-one — `i`n, `o`ut, `l`evel, `m`in/`M`ax time):

| key | action |
|---|---|
| `i` | add filter-**IN** (prompt; pre-filled from cursor's fields) |
| `o` | add filter-**OUT** (prompt; pre-filled from cursor's fields) |
| `l` | add **level** filter (picker over `kind` ∈ tool_call/error/rate_limit/hook/status) |
| `m` | add **min-time** filter |
| `M` | add **Max-time** filter |
| `\` | open / focus the filter panel |
| `f` | master switch — disable all filtering (toggle); status line shows `filtering OFF` |

Predicate syntax for `i`/`o` prompts: `agent:cc-1`, `tool:Bash`, `kind:tool_call`, `error:1`, `parent:toolu_xxx`, or `re:<regex>` for the rendered oneline.

In-panel hotkeys (when panel has focus — disambiguates from entry-view jumps):

| key | action |
|---|---|
| `Space` | enable/disable focused filter |
| `Enter` | edit focused filter's predicate |
| `t` | flip **IN ↔ OUT** on focused filter (lnav's `t: To OUT`) |
| `D` | delete focused filter |
| `i` / `o` / `l` / `m` / `M` | add new (same as global) |
| `f` | master off (same as global) |
| `Esc` | close panel; filters stay applied |

Quick-filter chords from cursor (no prompt, prepend to stack):

| chord | action |
|---|---|
| `=a` / `\a` | filter-IN / filter-OUT by **agent** of cursor |
| `=t` / `\t` | filter-IN / filter-OUT by **tool name** |
| `=k` / `\k` | filter-IN / filter-OUT by **kind** |
| `=p` / `\p` | filter-IN / filter-OUT by **parent_tool_use_id** |

Command-line equivalents (`:` from inspector): `:filter-in agent=cc-1`, `:filter-out tool=Bash`, `:filter-clear`, `:filter-level error,tool_call`, `:filter-time 12:34..12:40`.

**Export-to-lnav escape hatch.** ac's Inspector covers the clipboard-integrated path. For deep ad-hoc analysis (SQL view, custom regex highlight, time-shift, the full lnav muscle memory) we ship two commands:

- `:export-log [path]` — writes the current (post-filter) activity-log buffer to `path` (default `$XDG_RUNTIME_DIR/ac/log-<session>.log`) using ac's lnav-compatible line format (timestamp + agent + kind + tool_name + tool_use_id + parent_tool_use_id + oneline; full output appended as continuation lines). The first export per session also drops a format definition at `~/.lnav/formats/installed/ac.json` if absent so lnav recognizes the columns.
- `:lnav` — runs `:export-log` to a temp path and `exec`s `lnav <path>` inside a sub-PTY pane that takes over the screen until the user quits lnav (`q`); ac then returns to the dashboard. No clipboard integration with lnav: this is read-only offline analysis; anything the user wants to keep gets pasted out via lnav's normal yank.

`:lnav` requires `lnav` on `PATH`; if missing, the command prints a one-line install hint and stays in ac. On Windows the export still works (the file is fine), but `:lnav` is a no-op unless `lnav` is reachable (e.g. via WSL or a native build).

### Agent event parser (`events.h`)

Pure C, no I/O, fully testable. Input: one top-level JSON object from the child's stdout stream. Output: zero or more `AgentEvent`s normalized into a backend-agnostic enum:

```
AgentEvent =
  | AssistantTextDelta { text }
  | AssistantTextDone
  | ThinkingDelta { text }                 // signature field stripped
  | ToolCallStarted { call_id, name, args_summary }
  | ToolCallFinished { call_id, ok, result_summary, duration_ms }
  | RateLimit { window, reset_at }
  | StatusChange { new_status }
  | Error { message }
  | TurnDone
```

Backend-specific JSON shapes are normalized in `events_cc.c` / `events_codex.c` and tested against captured fixtures stored under `tests/fixtures/`.

**cc protocol notes (verified against `/home/amakrushin/proj/nx/out.json`).**
- Stream is **NDJSON** — one top-level JSON object per line on real stdout. Captures piped through `jq .` are pretty-printed concatenated JSON; both parse the same way with `cJSON_ParseWithLengthOpts` (parse one object, advance the cursor, repeat). The parser must accept both forms transparently.
- Event types we handle: `system/init` (capture model name, session_id, tool list — display as agent metadata), `system/hook_started` + `system/hook_response` (compact one-line entries in the activity log), `assistant` (walk `message.content[]`: `text` → `AssistantTextDelta`, `tool_use` → `ToolCallStarted` with `id`/`name`/`input`, `thinking` → `ThinkingDelta` after stripping the ~20 KiB `signature` field), `user` (walk `message.content[]`: `tool_result` → `ToolCallFinished` matched by `tool_use_id`, `is_error` flag drives `ok`), `rate_limit_event` → `RateLimit`, `result` → `TurnDone`.
- Thinking signatures: drop on parse; never store, never log, never paste. They're internal Claude bookkeeping with no user value.
- `parent_tool_use_id` indicates a sub-agent invocation (e.g., `Task` tool); v1 logs these inline, doesn't try to nest visually.

**Permission mode for v1.** Spawn cc with `--permissionMode dontAsk` so it never blocks waiting for user approval (the captured run was in this mode; tools that needed approval gracefully degraded). Document the limitation in `--help`. v2 adds a permission-forwarding modal that surfaces approval prompts in the dashboard.

### Curation

Pure C, no UI deps, fully unit-tested. Operates over an **agent's transcript**, not raw bytes.

- `Hunk` = `{ id, source_agent_id, transcript_span, text }`.
- **Splitter** modes:
  - default: per-message paragraphs (blank-line delimited within a single transcript message).
  - re-split: per-line.
  - manual: user-defined start/end.
- **Staging**: per-agent list of pending hunks.
- **Clipboard**: append-only ordered list of accepted hunks across all agents. Each entry knows origin (`cc-1`, `codex-2`). `d` revokes.
- **Paste**: serialize and write to any agent's stdin as a user message (default: each hunk on its own block, `# from cc-1:` origin marker; toggle markers off with a flag).

### Platform abstraction

Tiny — agent I/O is just pipes:

- POSIX: `pipe2` + `posix_spawnp` (or fork/exec) + `read/write` on the pipe FDs + `waitpid`.
- Windows: `CreatePipe` + `CreateProcessW` with redirected handles + `ReadFile/WriteFile` + `WaitForSingleObject`.

A `proc.h` interface with `proc_spawn` / `proc_read_stdout` / `proc_write_stdin` / `proc_kill` / `proc_wait` and two impls. No PTY required for the agent backends in v1. (PTY layer is deferred along with `sh` backend.)

## Dependencies

All vendored as git submodules under `third_party/`, compiled directly from `build.zig`.

- **termbox2** — single-header TUI lib. Public domain.
- **cJSON** — single-pair C JSON parser/printer. MIT.
- **greatest** — single-header C test framework. Public domain.

That's it. Everything else is `<stdlib.h>` / `<string.h>` / `<threads.h>` / `<unistd.h>` / `<windows.h>` etc.

Toolchain: `zig cc` from Zig 0.14.x (clang ~18), pinned in `.zigversion`. C standard: `-std=c2x` when probed via `__STDC_VERSION__`, else `-std=c17`.

## Testing strategy

The TUI is glue. The hard logic lives below it (event parsing, agent state machine, curation, bus). Push test coverage as far down the stack as possible; the TUI itself gets the lightest tier.

**Tier 1 — model unit tests (must-have for v1).** Pure-C tests with `greatest`, no I/O, no threads.
- `test_events_cc.c` / `test_events_codex.c`: replay captured NDJSON fixtures (`tests/fixtures/cc/*.ndjson`, `tests/fixtures/codex/*.ndjson`) → assert the produced `AgentEvent` sequence. Fixtures are captured once from real cc/codex sessions and committed; replays are fully deterministic.
- `test_curation.c`: splitter modes, staging accept/discard, clipboard accept/revoke, paste serialization (origin markers on/off).
- `test_ring.c`: byte/object ring roll-over.
- `test_bus.c`: append from N writer threads, tail-read from one reader, no torn entries (run with TSan in CI).

**Tier 2 — scenario tests (must-have for v1).** Wire real `events_*` + `agent` + `bus` together with a **fake `proc` backend** (`proc_fake.c`, behind `-DAC_PROC_FAKE`) that replays a fixture instead of `posix_spawn`. Drive a few full scenarios: agent thinking → tool call → tool result → done; agent error mid-stream; concurrent two-agent timeline merge in the bus. Asserts on bus log entries and per-agent transcript shape. **No TUI involved** — this catches 90% of integration bugs at zero TUI cost.

**Tier 3 — TUI rendering snapshot tests (stretch for v1, deferred otherwise).** Termbox2 has an in-memory backbuffer (`tb_cell_buffer`). Add a `render_to_string()` helper that flattens the backbuffer to a fixed-width unicode grid plus a parallel attr grid. Per-view tests: feed a known state into `ui_dashboard_render(state, &screen)` etc., dump → diff against a committed `.snap` file. Update snaps with `UPDATE_SNAPSHOTS=1`. Won't catch input-handling bugs but freezes layout regressions cheaply. **Decision rule: add Tier 3 once a view's layout has stopped churning. For v1 ship without it; expect to add it for `ui_dashboard_render` first because that's the most layout-stable view.**

**Tier 4 — end-to-end driving (out of scope for v1).** Spawn the real `ac` binary under a pty harness (`pexpect`-style shell or Python script under `tests/e2e/`), fake agents via stub backends, send keystrokes, screenshot, diff. Most fragile, slowest to run; defer until a flow has burned us at least once.

**CI matrix for v1.** Three OS targets × Tier 1+2 every PR. Tier 3 added per-view as it stabilizes. Tier 4 not run.

**Practical implications baked into the build order.**
- Step 1 (validation spike) lands fixtures into `tests/fixtures/cc/`. They become the inputs for Tier 1 from then on.
- The `proc_fake.c` from Tier 2 lives under `tests/` and only links into the test binary — it does *not* leak `#ifdef`s into production.
- The `agent.c` API takes a `Proc*` interface, not a concrete `posix_spawn` call. Same with `bus.c` (it takes a clock function, so tests can fake time).

## Verification

- `zig build && zig build test` green on Linux. macOS via local build. Windows via `zig build -Dtarget=...` + a Windows runner.
- Launch `ac`. `:spawn cc` and `:spawn codex` (twice) — three agent buttons appear at the top.
- Send a small task to each (drilldown view, type, Esc back). Activity log shows interleaved tool calls from all three with timestamps and agent ids.
- Hit **F7** from dashboard → Inspector opens with all entries expanded inline (args + output).
  - `Tab` jumps tool-to-tool; `Ctrl+→` jumps within one agent's tool calls; `Alt+→` jumps among sub-agent (`Task`) tool calls if any are present.
  - `v`, extend down 3 entries with `j`, `y` — selected tool outputs land in ac's clipboard. Esc back to dashboard, drill into another agent, `Ctrl+P`: yanked outputs are sent as input.
  - `Y` instead — content lands in the system clipboard (verify by pasting into another terminal).
- Drill into cc, `Ctrl+G` → curation mode. Three or four hunks. Accept two with `y`, discard the rest with `n`, `q`.
- `Ctrl+B` shows the clipboard with two accepted hunks.
- Drill into codex, `Ctrl+P` pastes the clipboard text as input. Verify codex sees it as a user message and starts working — visible as a new tool call in the dashboard activity log.
- `e` on a hunk → `$EDITOR` opens, edit, save → edited version lands in clipboard.
- `Ctrl+B`, select an old entry, `d` → it's gone. Subsequent `Ctrl+P` no longer includes it.
- `Ctrl+W` on a focused agent kills it cleanly; button changes to "killed", agent removed from the active set.
- Resize the terminal — dashboard reflows, drilldown wraps text properly.
- Repeat the core flow on macOS and Windows to confirm parity.

## Out of scope for v1

- The `sh` / generic-shell backend (deferred — needs a real PTY layer that we don't need yet).
- **Top menu bar** (mc-style `Left File Command Options Right` pulldowns) — F9 slot is reserved in the bar but the menus themselves are v2.
- **Unified agent-instructions config** that `ac` owns and projects to `CLAUDE.md` / `AGENTS.md` per backend. v1 lets each backend self-discover from its working directory.
- Daemon / persistence after terminal close.
- Any LLM brain inside `ac`.
- Permission-prompt forwarding from cc (v1 runs cc with `--permissionMode dontAsk`; tools that need approval gracefully degrade).
- Auto-summarization / token counting.
- Network, multi-user, auth.
- Saving/restoring sessions to disk.

## Open questions deferred to during implementation

- Exact codex stream-json flags / event shapes (codex's headless protocol changes more than cc's; pin to a specific codex version).
- Activity-log entry granularity for fast tool calls (`grep` returning instantly): collapse consecutive same-tool calls? First cut: don't collapse, see how noisy it gets.
- Whether the parser should reconstruct full content blocks before emitting (cleaner) or stream deltas (more responsive). First cut: stream `AssistantTextDelta` + `ToolCallStarted/Finished`; reconstruct on the UI side for the transcript.
- Tool-call argument summarization for the log row — pretty-print first N chars of the JSON args, escape newlines. Refine after observing real logs.
- C11 threads (`<threads.h>`) availability across all three platforms; fallback to pthread/Win32 shim if patchy.
